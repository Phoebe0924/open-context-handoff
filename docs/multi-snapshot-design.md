# Multi-Snapshot 设计（v2 方向，主线+子线模式）

> 状态：设计已确认，尚未实现。
> 触发原因：单一 Snapshot 的 300 字上限，在涉及多条并行执行线的复杂任务中
> 不够用（例如：同一客户案例同时承载"诊断进度"、"方法论沉淀"、"内容素材
> 积累"三条独立线索）。
> 本文档不是对 300 字限制的放宽，而是承认：当 300 字真的装不下时，说明
> 这不是一个 Snapshot，而是被压缩进一个文件里的好几个 Snapshot。

---

## 设计原则

300 字限制本身不变，也不应该变。它的价值正是在于"逼迫只留下不能丢的东西"。
如果某个任务持续撞到这条线，正确的回应不是放宽限制，而是拆分任务的描述
单元——用多个 Snapshot 取代一个超载的 Snapshot。

子线的存在不能制造新的"解释成本"。接收方不应该需要自己判断"这几份文件
该怎么拼起来"——这件事必须由主线显式告知，否则就违背了 OCH 最初要解决
的问题。

---

## 结构

### 单线/多线判定

- 由模型自动判断，不要求用户提前选择。
- 判定标准：对话中如果出现两条以上目标不同、可分别维护的任务线索，
  即使共享输入或下一步，也判定为多线；否则为单线。
- 判定结果必须可验证，不能是黑盒：PART 2 审核备注第一行固定格式为
  "判定为单线/多线，理由：……"，单行不超过50字，和 PART 2 其他条目
  规则一致。

### 主线 Snapshot

- 保留原有六字段结构：WHAT WE ARE DOING / CURRENT STATE / COMPLETED /
  DECISIONS / CONSTRAINTS / NEXT ACTION。
- 300 字总量限制不变。
- 新增一项：**子线清单**，位置在 NEXT ACTION 之后（六字段结构本身顺序
  不变，子线清单作为第七项追加在最后）。理由：接收方应先完整读完主线
  在讲什么、要做什么，再看"还有哪些细节可查"——子线清单是索引，不应该
  打断主线信息的连贯阅读。列出当前存在哪些子线，每条子线一行，包括子线
  名称和一句话描述其覆盖范围。子线清单计入主线 300 字总限制（已确认），
  这样能保住 300 字限制本身的约束力，避免清单变成绕开精简要求的逃生
  通道。
- NEXT ACTION 规则扩展：如果接下来要做的事依赖某条子线的内容，NEXT
  ACTION 必须显式标注"先读 XX 子线"。如果不依赖任何子线，无需标注。

子线是否需要被读取，完全由主线的 NEXT ACTION 决定，不依赖接收方自行
判断。这是整个设计里最关键的一条规则——它让"按需读取"变成可验证的，
而不是一句空话。

### 子线 Snapshot

- 不设字数上限。子线本质上是从主线拆出来的详情，它存在的意义就是承载
  主线装不下的内容，所以不应该再被字数卡住。
- 字段简化为三项，不使用完整六字段，也不包含 NEXT ACTION：
  - **这条线在讲什么**（对应主线 WHAT WE ARE DOING 的子集）
  - **目前到哪了**（对应 CURRENT STATE + COMPLETED 的合并）
  - **已确定的东西**（对应 DECISIONS，只保留这条线内部的决定）
- 不设 NEXT ACTION 字段的原因：所有"接下来做什么"的调度权统一收在主线，
  子线如果各自维护一个 NEXT ACTION，会出现多头指挥、相互矛盾的风险。

### 文件组织规则

- **默认行为**：所有子线和主线放在同一个文件里，用标题分区
  （如 `## 子线：XXX`）。这是默认值，不需要额外判断。
- 模型不自行拆分文件，只提示："如果某条子线需要被反复独立跟进，建议拆成
  独立文件。"是否拆分由用户决定。
- **例外情况**：如果某条子线会被反复、独立地多次重新打开和更新（例如
  一个需要持续跟进数周的客户项目线），才把这条子线拆成独立文件。
- 判断标准只有一条：**这条子线是否会被单独反复读取/更新**。是 → 独立
  文件；否 → 留在主文件里分区。不需要在动笔前提前规划，子线通常是在
  长对话结束、回看时才被发现的，所以这条规则本身允许"先合并、后拆分"。

## 一个例子（结构示意，非真实数据）

```
# Snapshot: 主线

### WHAT WE ARE DOING
...

### NEXT ACTION
先读子线A，然后……

### 子线清单
- 子线A：诊断进度
- 子线B：方法论沉淀

---

## 子线：诊断进度

**这条线在讲什么**
...

**目前到哪了**
...

**已确定的东西**
- ...

---

## 子线：方法论沉淀

**这条线在讲什么**
...
```

## 还没解决的问题

- 子线数量上限要不要设？目前没有讨论。如果一份主线下出现 5 条以上子线，
  这件事本身可能说明任务粒度划分有问题，而不是协议该兼容更多子线。
- 子线之间是否允许互相引用？当前设计是不允许的——子线只对主线负责，
  子线之间不互相感知。如果未来发现子线之间确实需要交叉引用，需要重新
  评估，因为这会显著提高复杂度。
